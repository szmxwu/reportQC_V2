# reportQC V3版本计划

## 目标重定义
- 单院区场景下，现有 V1/V2 规则方案已经足够可用，V3 不以提升单院区效果为核心目标。
- V3 要解决的核心问题是多院区泛化：不同院区在报告模板、术语习惯、句式风格、申请单书写方式、设备命名、缩略表达上的差异，导致现有规则方案迁移成本高、维护成本高、上线速度慢。
- 因此，V3 的目标不是“用 LLM 替换全部规则”，而是构建一个面向多院区泛化的混合质控系统：规则保留强确定性能力，领域模型承担复杂语义判断，必要时由更强 LLM 复核。

## 总体原则
- 利用 reportQC V2 现有能力、百万级历史放射报告数据和本地 LLM，构建可持续迭代的数据生产体系。
- 训练一个 4B 级领域模型作为 V3 主判模型，在保证实时性的前提下，提升跨院区泛化能力。
- 强规则项继续保留，不为“模型化”而模型化。
- 复杂语义项逐步迁移到领域模型，但必须保留结构化输出、可追溯证据和置信度。
- 上线策略采用 V2/V3 并行灰度，不直接替换现网规则系统。

## 问题定义

### 当前系统的优势
- 单院区规则适配充分，V1/V2 在固定报告风格下已能满足大部分质控需求。
- 实体抽取、知识图谱、术语字典、危急值规则、语法检测和 LLM 后置精筛已经形成较完整的工程基础。
- 当前代码已经具备“自动打标签器”的雏形，可用于 V3 数据生产。

### 当前系统的核心瓶颈
- 规则泛化能力弱：不同院区的报告风格、模板和表达习惯差异很大，规则无法穷尽。
- 新院区接入成本高：每个院区都要维护 replace 规则、术语字典、忽略项和阈值配置。
- 通用 LLM 不懂放射语料：放射报告是闭源专业数据，在开源模型预训练语料中占比极低。
- 复杂语义任务难以完全规则化：尤其是描述与结论一致性、方位关系、申请单对照、复杂语法和危急值进展判断。

### 对放射报告文本特点的补充判断
- 放射报告相比电子病历，依然是一种语法相对简单、结构较稳定的文本。
- 其核心骨架通常可以概括为“部位 + 描述”，结论部分则是对描述部分的压缩和归纳。
- 多院区差异的核心不在于开放域复杂语义，而主要在于两类问题：
  1. 解剖部位粒度差异：专科医院通常会关注更多细小、少见、专病相关的解剖结构，综合医院则更偏向粗粒度部位表达。
  2. 描述习惯差异：不同院区会用完全不同的句式描述同一个医学事实。例如“胸腺未见增大”和“前纵隔见帆状软组织密度影”表面风格差异很大，但在特定人群和语境下都可能在描述儿童正常胸腺。
- 因此，V3 的泛化重点不是让模型解决特别复杂的长链推理，而是让模型学会“多风格表达映射到同一医学语义”的能力。

## 可行性分析

### 原则上是否可行
- 可行。
- 但真正的关键不是“是否能训一个 4B 模型”，而是能否构建高质量、多院区、低噪声的训练数据体系。
- 如果直接拿 V2 全量规则输出当作金标准，V3 大概率只会学会复述 V2 的局限，无法真正解决多院区泛化问题。

### 为什么 V3 适合做多院区泛化
- 多院区差异更多体现在语言表达、模板风格、隐含语义和书写习惯，而不是医学常识本身。
- 这些差异正是领域模型比纯规则更擅长吸收和迁移的部分。
- 当前项目已有实体抽取和规则系统，可作为稳定底座和银标数据来源，不需要从零开始构建训练体系。

### 额外可行性判断
- 放射报告的语言空间虽然比电子病历更收敛，但其跨院区差异高度集中在“表达映射”和“解剖粒度变化”上，这恰好适合做领域模型吸收和迁移。
- 换句话说，V3 的主要收益不会来自模型替代全部规则，而会来自模型对跨院区异构表达的归一化能力。

## 实体抽取层在 V3 中的定位

### 当前实体抽取层的价值
- 当前项目的底层是实体抽取层。
- 规则抽取层的真正优势，不只是识别出文本中的实体，而是在抽取的同时完成标准化：
  - 通过 partlist 把文本中的解剖部位锚定到知识图谱中的唯一位置。
  - 快速确定实体之间的上下级关系、父子关系和方位关系。
  - 为后续 conclusion_missing、orient_error、contradiction、apply_orient 等任务提供统一结构支点。
- 对本项目来说，解剖实体是后续质控任务最关键的基础能力之一。

### 当前规则实体层的局限
- 规则实体层的主要问题不是精度不够，而是维护成本高。
- 一旦新院区出现新的同义词、近义词、缩写、专科术语或风格化表达，就需要持续补规则。
- 这部分工作正是多院区接入成本居高不下的根源之一。

### 这部分工作能否被 LLM 替代
- 结论：不能被纯 LLM 完全替代，但可以被显著增强。
- 不建议把底层实体抽取完全改为纯 LLM，原因如下：
  1. 实体抽取不仅要“识别词语”，更要“稳定地映射到标准知识图谱节点”。
  2. partlist 所承载的是确定性锚定和层级关系，这要求结果稳定、可复现、可审计。
  3. 如果实体层完全变成自由生成，后续规则和模型任务都会失去稳定的结构基础。

### 推荐路线：规则实体层 + 模型增强归一化
- V3 中更合理的路线是“规则主导 + 模型增强 + 图谱约束回写”，而不是“纯 LLM 实体抽取”。
- 推荐拆成三层：
  1. 第一层：规则与词典抽取。继续承担高精度识别和知识图谱锚定。
  2. 第二层：模型候选补充。对规则未命中或低置信实体，使用领域模型补充候选标准部位或同义表达。
  3. 第三层：图谱约束回写。模型输出必须回写到标准节点体系，最终仍生成稳定的 partlist 或标准节点 ID。
- V3 对实体层的目标不是取消 partlist，而是降低人工维护 partlist 映射规则的成本。

## 能力边界划分

### 需要迁移到领域模型主判的功能
- partinverse：检查项目方位错误
- conclusion_missing：结论与描述不对应
- orient_error：结论与描述方位不符
- contradiction：语言矛盾
- apply_orient：申请单与报告方位不符
- grammer_error：复杂语法错误

### 不建议迁移到领域模型的功能
- partmissing：报告部位缺失，强规则更稳定
- special_missing：检查方法错误或缺失，规则即可
- sex_error：性别错误，规则即可
- measure_unit_error：测量单位错误，规则即可
- none_standard_term：术语不规范，字典维护更可靠
- RADS：关键词和规则检测即可

### 需要规则 + 模型联合判断的功能
- Critical_value：规则负责危急征象召回，领域模型判断是否新发、进展、持续加重以及是否需要升级为危急值

## V3 架构定位

### 目标架构
1. 规则层
负责结构化、确定性、低歧义任务，作为稳定底座，同时保留实体抽取和知识图谱锚定能力。

2. 领域模型层
使用 4B 级模型作为主判模型，承担复杂语义任务，同时为实体层提供跨院区表达映射和候选归一化能力，输出结构化 JSON 结果。

3. 强 LLM 复核层
仅对低置信、高风险、跨院区新风格样本进行复核，用于兜底和持续产数。

### 核心设计思想
- V3 不是纯 LLM 系统，而是“规则产数 + 领域模型主判 + 强规则兜底 + 大模型复核”的混合系统。
- 单院区不必强制切 V3 主链路；V3 首先服务新院区接入和多院区统一化。

## 实施路线图

### 阶段 0：准备期
目标：明确 V3 成功标准，建立跨院区问题定义和数据基线。

输出物：
- 院区清单及数据权限边界
- 不同院区的报告模板差异分析
- 各院区任务分布统计
- 多院区验证集定义

关键动作：
- 选取至少 3 个风格差异明显的院区作为第一批泛化验证院区。
- 梳理每个院区的报告描述、结论、申请单、检查条目命名规范和字段质量。
- 定义哪些任务以跨院区泛化为第一优先级，建议优先是 conclusion_missing、orient_error、contradiction、apply_orient、grammer_error。

验收标准：
- 已形成“院区差异地图”。
- 已形成跨院区 holdout 集合，不能用于训练。

### 阶段 1：V2 数据生产体系建设
目标：把 V2 从推理系统升级为 V3 的银标数据工厂。

输出物：
- 任务级银标数据集
- 高置信阳性样本池
- 高置信阴性样本池
- 规则冲突样本池
- LLM 精筛样本池

关键动作：
- 用现有 V2 能力批量生产任务级样本。
- 对不同任务定义不同的高置信规则，不允许“一套规则打所有标签”。
- 对多院区数据按院区分桶，保留来源信息，防止训练时丢失域信息。
- grammar_error 单独建数据流，不与结构一致性任务混训同标签。
- 单独沉淀实体相关样本：实体未命中样本、低置信标准化样本、院区新增解剖表达样本、多种表达对应同一标准部位的样本。

验收标准：
- 每个核心任务均有可训练样本。
- 每个院区均有可用训练样本和独立验证样本。
- 样本中保留院区 ID、模态、部位、任务名、证据句、标签来源、置信等级。

### 阶段 2：数据清洗与人工校准
目标：降低伪标签噪声，避免模型学习到规则误差。

输出物：
- 任务级高质量训练集
- 人工校准小金标集
- 多院区评测基准集

关键动作：
- 对高价值任务做分层抽样人工校验。
- 优先人工校验跨院区差异最大、规则最容易漂移的任务。
- 使用本地强 LLM 对规则样本进行二次判别，只保留规则和 LLM 一致或 LLM 高置信支持的样本。
- 将“规则阳性 + LLM否定”“规则阴性 + LLM高置信阳性”单独存储，作为 hard case 集合。
- 对实体层建立小规模金标集，重点覆盖专科细粒度解剖结构、缩写、近义词、风格化描述和需要映射回标准解剖节点的句子。

验收标准：
- 每个核心任务至少有一批人工核验样本。
- 高置信训练集 precision 达到可控区间。
- 已形成单院区内验证集和跨院区验证集两套基准。

### 阶段 3：领域适配训练
目标：让底座模型先学会放射报告语言，而不是直接学习任务格式。

输出物：
- 领域适配后的 4B 基础模型
- 院区风格对齐语料集

关键动作：
- 用多院区原始报告做继续预训练或轻量持续训练。
- 覆盖描述、结论、申请单、检查条目、模板句式、常见缩略词、部位表达和否定表达。
- 多院区混合采样时要控制院区比例，避免大院区数据压制小院区风格。
- 显式纳入“同一医学语义的多种院区表达”语料，让模型学习跨院区表达映射，而不是只学习模板字面形式。

验收标准：
- 模型在放射报告语料上的困惑度明显下降。
- 领域词汇、方位词、否定句和模板化句式理解能力提升。

### 阶段 4：任务监督微调
目标：让领域模型学会输出结构化质控结果。

输出物：
- V3 主判模型
- 统一结构化输出 schema
- 实体归一化增强子能力

关键动作：
- 输入统一包含：ReportStr、ConclusionStr、StudyPart、Sex、applyTable、modality。
- 输出统一为结构化 JSON，字段包括：任务名、是否命中、证据句、错误短语、建议修正、置信度。
- 对任务进行多任务联合训练，但 grammar_error 可作为单独头或单独模型实验。
- 训练中显式加入院区标识或风格标签，帮助模型学习域差异。
- 实体层单独做两类实验：
  - 实验 A：模型直接输出标准解剖节点候选。
  - 实验 B：模型仅做候选召回，最终由图谱约束完成标准化。
- 默认优先考虑实验 B，因为其稳定性和可控性更高。

验收标准：
- 多院区验证集上，核心任务优于 V2 或至少明显降低跨院区性能塌陷。
- 结构化输出稳定、可解析率高。

### 阶段 5：联调与灰度上线
目标：在不影响现网单院区稳定性的前提下，将 V3 接入多院区场景。

输出物：
- V2/V3 共存策略
- 院区级路由开关
- 低置信复核策略

关键动作：
- 单院区继续默认 V2。
- 新院区或泛化较差院区优先走 V3 shadow 模式。
- 对 V3 低置信样本启用更强 LLM 或人工复核。
- 将 V2 与 V3 差异输出沉淀为下一轮训练数据。

验收标准：
- 新院区接入周期缩短。
- V3 在新院区优于直接套用 V2 默认规则。
- 单院区现网不受影响。

## 数据流设计

### 数据来源
1. 多院区历史正式报告
2. 申请单文本
3. 审核前后差异报告
4. V2 规则输出结果
5. 本地 LLM 复核结果
6. 人工审核样本
7. 实体抽取失败样本、低置信标准化样本和新增术语样本

### 数据分层
1. 原始语料层
保存各院区原始报告、申请单、检查信息和时间信息。

2. 结构化抽取层
由 V2 实体抽取和规则模块生成结构化实体、任务候选和证据句，并保留实体是否命中知识图谱、是否标准化成功、是否存在候选歧义等信号。

3. 银标层
按任务生成高置信阳性、高置信阴性、冲突样本和 hard case。

4. 金标层
由人工和高质量 LLM 联合校准形成小规模高质量基准集。

### 实体层专用数据流
1. 规则命中且标准化成功的实体，作为高置信正样本。
2. 规则未命中但人工确认存在实体的样本，作为实体召回补充样本。
3. 规则命中但标准化错误的样本，作为实体纠错样本。
4. 跨院区不同表达指向同一标准节点的样本，作为归一化样本。

### 多院区数据组织原则
- 所有样本必须携带院区 ID。
- 训练集、验证集、测试集都要保留院区分布信息。
- 必须设置跨院区 holdout，不能只看随机切分指标。

## 训练流设计

### 训练路线
1. 领域继续预训练或轻量持续训练
2. 多任务监督微调
3. 实体归一化增强训练
4. 按任务做精调或蒸馏
5. 低置信样本增量回流再训练

### 训练策略建议
- 单院区验证阶段：可以先不做继续预训练，只做 LoRA 验证流程。
- 多院区泛化阶段：建议至少做一轮领域适配，再做 LoRA 微调。
- grammar_error 建议独立实验：
  - 方案 A：与其他任务联合训练
  - 方案 B：独立语法模型
  - 最终按效果选型
- 实体层建议单独作为“归一化增强子任务”设计，不要默认它会在上层任务训练中自然学会。
- 实体层的目标不是生成解释文本，而是稳定输出标准节点候选或标准部位标签。

### 资源评估
- 仅 LoRA：1 到 2 张 24GB 显存卡可做首轮验证。
- 继续预训练 + LoRA：建议 2 到 4 张 24GB 到 48GB 卡。
- 如果上下文较长或做多任务联合训练，资源预算需进一步上浮。

### 样本规模建议
- 首轮不追求百万级全量训练。
- 推荐先做 5 万到 20 万条高质量样本。
- 其中多院区样本占比必须平衡，避免模型只学到大院区风格。

## 评估流设计

### 评估目标
- 核心不是提升随机切分分数，而是降低跨院区性能塌陷。

### 评估维度
1. 任务级 precision、recall、F1
2. 证据句命中率
3. 关键误报率
4. 高风险漏检率
5. 跨院区泛化差值
6. 推理时延和吞吐
7. 实体标准化准确率
8. 跨院区同义表达映射准确率

### 必须分开的评估集
1. 单院区验证集
用于确认不弱于现有 V1/V2。

2. 跨院区验证集
用于观察泛化能力。

3. 新院区冷启动测试集
模拟真实上线前的新院区接入效果。

### 关键指标建议
- conclusion_missing、orient_error、contradiction、apply_orient：重点看 F1 和关键误报率。
- grammer_error：重点看 precision、句级证据准确率和误报类型分布。
- Critical_value：重点看高风险漏检率。
- 多院区泛化：重点看最差院区表现，而不是平均值。
- 实体层重点看三项：实体召回率、标准化准确率、新院区新增表达映射成功率。

## 上线灰度方案

### 上线原则
- 不替换现有单院区稳定链路。
- 优先解决新院区和跨院区统一部署问题。
- 所有切换都必须可回滚。

### 灰度路径
1. Shadow 模式
V3 与 V2 并行输出，不影响现网，只做差异分析。

2. 半自动模式
V3 仅对多院区复杂任务给出建议，仍由 V2 或人工主判。

3. 院区级切换
对泛化问题最明显的新院区，优先启用 V3 主判。

4. 低置信兜底
V3 低置信时回退到 V2 或调用强 LLM 复核。

### 运营反馈闭环
- 所有 V2/V3 差异样本都进入回流池。
- 按院区统计误报、漏报和低置信分布。
- 每月或每季度做一次增量训练或规则补丁更新。

## 与当前项目的衔接点
- [NLP_analyze.py](NLP_analyze.py) 仍可作为 V2 总编排入口和数据生产入口。
- [api_server.py](api_server.py) 可承载 V2/V3 共存接口和灰度开关。
- [Extract_Entities.py](Extract_Entities.py) 提供稳定的结构化抽取基础，可继续作为规则底座。
- [report_analyze/report_conclusion_checker.py](report_analyze/report_conclusion_checker.py) 和 [report_analyze/contradiction_checker.py](report_analyze/contradiction_checker.py) 可作为银标生成器。
- [report_analyze/llm_validator.py](report_analyze/llm_validator.py) 可演化为数据清洗和 teacher 组件。
- [grammer/inference/detect_real_data_final.py](grammer/inference/detect_real_data_final.py) 可独立承担 grammar 子任务的数据生产与评估。

## 主要风险点
1. 伪标签噪声过大，模型学到的是 V2 误差而不是真实规律。
2. 多院区数据不均衡，模型被大院区风格主导。
3. grammar_error 任务边界过宽，导致训练目标不稳定。
4. 如果实体层被过早完全模型化，会失去当前知识图谱锚定和层级关系的稳定优势。
5. 只看平均指标，掩盖了个别院区性能很差的问题。
6. 过早替换 V2，影响现有单院区稳定链路。

## 当前建议结论
- V3 的成功标准不是“单院区比 V1 更强”，而是“新院区接入更快、跨院区效果更稳、对规则依赖更低”。
- 第一阶段优先解决多院区泛化，不追求所有任务一次性模型化。
- 实体抽取层不建议被纯 LLM 完全替代，而应演进为“规则主导 + 模型补充 + 图谱约束回写”的混合层。
- 建议先在 3 个以上风格差异明显的院区上做验证，证明 V3 在跨院区上明显优于直接复制 V2 规则配置，再逐步扩展。

## 子计划一：规则实体层 + 模型增强归一化自动化流水线

### 1. 子计划定位
- 这是 V3 的第一份子计划书，也是整套 V3 体系的地基工程。
- 目标不是直接提升某一个质控任务指标，而是建设“多院区可迁移的实体标准化基础设施”。
- 如果这一层做不好，后续的 conclusion_missing、orient_error、contradiction、apply_orient 乃至 Critical_value 都会失去统一的结构支点。

### 2. 现有基础回顾

#### 2.1 当前实体抽取主链
- 当前实体抽取主入口是 [Extract_Entities.py](Extract_Entities.py#L663) 的 `text_extrac_process()`。
- 现有处理链可概括为：
  1. 预处理文本 `preprocess_text`
  2. 基于统一 FlashText 抽取器做关键词匹配 `Extract_Keywords`
  3. 按实体位置重切短句 `resplit_sentence_by_entities`
  4. 方位抽取 `extract_orientation`
  5. 疾病描述抽取 `get_all_illness_descriptions`
  6. 数值与阳性处理 `process_positive_batch`
  7. 歧义消解 `disambiguate_entities`
  8. 部位合并 `merge_part`
  9. 原始短句匹配 `match_short_sentence_batch`
  10. 病理属性抽取 `get_pathology_attribute`
- 其中最关键的产物是 `partlist`，它把文本中的部位映射为知识图谱中的父子路径。

#### 2.2 当前知识底座
- 报告内容知识图谱：[config/knowledgegraph.xlsx](config/knowledgegraph.xlsx)
- 检查部位知识图谱：[config/knowledgegraph_title.xlsx](config/knowledgegraph_title.xlsx)
- 抽取器构建入口：[Extract_Entities.py](Extract_Entities.py#L162) 的 `build_unified_processor()`
- 当前方案已经具备三种重要能力：
  1. 同义词归并
  2. 父子部位层级表达
  3. 基于坐标轴的歧义消解前置信息

#### 2.3 当前瓶颈
- 现有体系强在“锚定稳定”，弱在“增量维护”。
- 最大问题不是精度，而是面对新院区时，新增同义词、缩写、细粒度解剖结构和风格化描述时，仍需要持续人工扩图。
- V3 在这一层的核心目标，不是推翻规则图谱，而是把“人工补图谱”升级为“自动化候选发现 -> 自动化归一化 -> 自动化回写审核”的流水线。

### 3. 目标定义

#### 3.1 最终目标
- 构建一套以现有知识图谱和规则抽取为锚点、以本地 LLM 为增强器、以自动化流水线为核心的数据生产与归一化系统。
- 使系统能够在尽量少依赖人工逐条维护的前提下，持续吸收多院区新增实体表达，并稳定回写到标准知识图谱节点。

#### 3.2 不做的事情
- 不把底层实体抽取直接改造成纯 LLM 生成系统。
- 不放弃 `partlist` 这类确定性结构输出。
- 不依赖人工逐条补知识图谱作为主路径。

#### 3.3 要达成的能力
1. 自动发现新院区实体表达
2. 自动生成标准节点候选
3. 自动判定候选是否可信
4. 自动回写为可复用的规则词表/候选库/知识图谱补丁
5. 自动评估新增表达对后续任务的收益

### 4. 核心思路

#### 4.1 总体原则
- 规则实体层继续承担“高精度锚定”。
- 本地 LLM 不直接替代实体层，而承担“候选发现、语义归一化、同义表达映射、低置信补充”的角色。
- 所有模型输出都不能直接进入主推理链，必须经过图谱约束和自动化验证后才能回写。

#### 4.2 流水线总览
建议把整套流水线拆成六段：

1. 新表达发现层
从多院区原始报告中自动发现规则未命中或低置信命中的候选实体表达。

2. 候选归一化层
使用本地 LLM 和图谱上下文，为每个候选表达生成标准节点候选。

3. 图谱约束层
根据现有知识图谱层级、检查部位上下文、模态、句法位置和共现模式，对候选做约束过滤。

4. 自动验证层
使用离线规则验证、跨语料频次验证、上下文一致性验证和下游任务收益验证，淘汰不可靠候选。

5. 回写产物层
将可信候选分别写入：
  - 实体同义候选库
  - 标准节点映射表
  - 院区特异别名字典
  - 知识图谱补丁候选表

6. 增量发布层
经过阈值控制后，将高可信产物纳入下一版规则抽取器和归一化器。

### 5. 自动化流水线设计

#### 5.1 阶段 A：新表达自动发现

##### 输入
- 多院区历史报告原文
- 当前规则抽取结果
- 当前 `partlist` 标准化结果
- 句子级上下文、检查部位、模态、申请单信息

##### 自动发现对象
1. 完全未命中的解剖实体表达
2. 命中了错误标准节点的表达
3. 只能命中粗粒度节点、但上下文显示应为细粒度节点的表达
4. 跨院区高频出现、但图谱中不存在的别名和缩写
5. 风格化描述中隐含的解剖表达

##### 发现策略
- 未命中候选挖掘：
  - 在带明显部位语义的短句中，寻找未被 `Extract_Keywords` 捕获的高频 n-gram
  - 联合词频、词性、上下文模板和句法位置筛选候选
- 低置信标准化挖掘：
  - 利用 `disambiguate_entities` 前后的歧义信息，沉淀多候选难例
- 下游失败反推：
  - 重点收集在 conclusion_missing、orient_error、apply_orient 中由于实体抽取失败而造成的错误样本

##### 输出
- `entity_candidate_discovery.jsonl`
- 字段建议包含：院区、原句、候选短语、上下文、模态、StudyPart、是否规则命中、现有候选节点、频次、共现信息

#### 5.2 阶段 B：本地 LLM 候选归一化

##### 目标
- 对候选表达生成一个或多个标准解剖节点候选，而不是直接让 LLM 给自由答案。

##### 提示输入设计
- 原始短句
- 前后句上下文
- StudyPart 和 modality
- 当前报告已抽出的其他实体
- 从知识图谱检索出的候选节点列表
- 候选节点的父子路径和同义词信息

##### 提示输出设计
- 输出必须结构化，包括：
  - 候选短语
  - 标准节点候选 1..N
  - 对应置信度
  - 归一化理由
  - 是否可能为新节点
  - 是否仅为现有节点别名

##### 关键约束
- LLM 不允许自由扩展到图谱外无限空间。
- 默认先做“候选排序”，而不是“自由命名”。
- 只有当所有现有节点都不合适时，才允许输出“疑似新节点”。

##### 输出
- `entity_candidate_normalized.jsonl`

#### 5.3 阶段 C：图谱约束与程序化校验

##### 校验逻辑
1. 层级一致性校验
- 归一化节点必须与 StudyPart、modality 和句中其他实体形成合理层级关系。

2. 语境一致性校验
- 候选节点必须与共现病理描述、方位词、检查部位和相邻实体兼容。

3. 院区频次校验
- 同一表达在单院区和跨院区的出现频次、上下文相似度、稳定性必须达到阈值。

4. 下游收益校验
- 将归一化后的实体重新送入现有 V2 任务链，观察是否减少 conclusion_missing、orient_error、apply_orient 等任务的异常。

5. 冲突校验
- 若新增归一化会与现有高置信映射冲突，默认不自动发布。

##### 产物分级
- A 级：可自动纳入别名库
- B 级：纳入候选库，待后续自动复核
- C 级：仅保留样本，不回写

#### 5.4 阶段 D：回写与发布

##### 回写目标
1. 院区特异别名字典
2. 通用同义词候选库
3. 标准节点映射补丁表
4. 新节点建议表

##### 不同类型的回写策略
- 已有节点新别名：优先回写到别名字典或知识图谱补丁表
- 已有节点隐式表达：优先回写到“模型候选库 + 图谱约束”层，不立即改主图谱
- 疑似新节点：进入“新节点候选表”，需要跨院区稳定性验证，不直接入主图谱

##### 发布机制
- 不直接改 `knowledgegraph.xlsx`
- 先生成自动补丁层，例如：
  - `config/knowledgegraph_alias_patch.xlsx`
  - `config/knowledgegraph_node_candidates.xlsx`
- 构建下一版统一抽取器时，同时加载主图谱和补丁层

### 8. 评估体系

#### 8.1 核心指标
1. 实体召回率
2. 标准化准确率
3. partlist 稳定率
4. 新院区新增表达吸收率
5. 实体层改造对下游任务的收益提升

#### 8.2 业务化指标
1. 新院区接入时人工补词数量
2. 单院区新增规则维护工时
3. 因实体抽取失败导致的质控误判占比
4. 每轮自动补丁发布后下游任务收益

#### 8.3 成功标准
- 新院区实体抽取适配不再主要依赖人工逐条补图
- 对新增表达，系统能自动给出高质量标准节点候选
- 自动回写机制能在可控误差下持续扩充实体层覆盖面
- 下游任务在新院区的泛化性能明显优于只复制旧规则配置

### 9. 里程碑建议

#### M1：候选发现跑通
- 能从多院区报告中自动生成未命中实体候选池

#### M2：LLM 归一化跑通
- 能基于上下文和图谱候选给出结构化标准节点候选

#### M3：图谱约束验证跑通
- 能自动筛出高可信别名候选和映射候选

#### M4：补丁层发布跑通
- 能在不手工改主图谱的情况下，将自动产出的高可信候选纳入抽取器

#### M5：下游收益闭环跑通
- 能证明实体层自动增强后，对 conclusion_missing、orient_error、apply_orient 等任务在新院区上有稳定收益

### 10. 风险与对策

#### 风险 1：LLM 候选漂移过大
- 对策：限制为“候选排序”优先，不允许直接自由生成最终节点

#### 风险 2：新增别名污染主图谱
- 对策：建立补丁层，分级发布，不直接修改主图谱文件

#### 风险 3：自动扩图对下游收益不稳定
- 对策：所有候选必须经过下游收益回放验证，再决定是否发布

#### 风险 4：专科细粒度节点过多，形成节点爆炸
- 对策：优先扩别名映射，不轻易扩主节点；新节点必须通过跨院区和多样本验证

### 11. 子计划结论
- 这份子计划书的核心，不是做一个“更聪明的实体抽取模型”，而是建设一套“自动发现 -> 自动归一化 -> 自动验证 -> 自动补丁发布”的知识增长流水线。
- V3 的实体层不应走“人工扩图”路线，也不应走“纯 LLM 自由抽取”路线。
- 最优路径是：
  - 用现有规则体系守住稳定锚定
  - 用本地 LLM 承担跨院区表达映射和候选补充
  - 用图谱约束和下游回放验证保证可控性
- 一旦这条流水线跑通，后续所有多院区泛化任务都会站在一个更稳定、更可扩展的基础之上。

## 价值杠杆重估

### 需要重新面对的事实
- 对质控任务而言，真实的严重放射报告错误本来就是低频事件。
- 如果 V3 的价值只定义为“比 V2 多抓住几个严重错误”，那么这套系统很容易陷入“杀鸡用牛刀”。
- 因此，V3 的最大价值不能只建立在“发现低频严重错误”这一点上，而必须建立在更高频、更可规模化、更能跨院区复用的能力上。

### V3 最大的价值杠杆不在于“查错”，而在于“统一和沉淀”
- 真正的高杠杆价值，是把不同院区风格各异的放射报告，统一映射到同一个结构化医学语义层。
- 一旦完成这个统一层，质控只是其中一个上层应用，而不是唯一应用。
- 换句话说，V3 最有价值的资产不是某个具体质控规则，而是：
  1. 多院区可迁移的解剖实体标准化能力
  2. 多风格描述到统一医学语义的映射能力
  3. 可持续迭代的银标/金标数据生产体系
  4. 可承接多种上层应用的放射结构化中间层

### 如果只做质控，ROI 可能不够高
- 因为严重错误低频，医院很难仅凭“多抓几个错误”持续为高复杂度系统买单。
- 但如果 V3 能把多院区报告统一成结构化、可计算、可检索、可运营的数据资产，那么它的价值会从“单点质控工具”上升为“放射数据基础设施”。
- 这时，质控只是最容易切入的第一个场景，而不是最终商业形态。

## 最有价值的商业化方向

### 方向一：多院区快速接入的放射质控中台
- 这是最直接、最现实的第一商业化路径。
- 核心卖点不是“AI 很强”，而是：
  - 新院区接入更快
  - 院区个性化规则维护更少
  - 多院区统一部署成本更低
  - 质控结果口径更一致
- 适合对接医联体、集团医院、区域影像中心和第三方影像平台。
- 对客户来说，真正有感知的价值是“减少适配人力和上线周期”，而不只是“提高若干精度指标”。

### 方向二：放射报告结构化与标准化引擎
- 这是比“质控”更大的商业化方向。
- 一旦系统能把不同院区、不同风格、不同模板的报告统一映射到标准结构化语义层，就可以对外提供：
  - 结构化报告转换
  - 解剖部位标准化
  - 异常发现标准化
  - 结论语义归一化
  - 多院区同质化数据治理
- 这类能力的商业价值通常高于单纯质控，因为它直接服务于数据治理、科研、随访、运营和平台化建设。

### 方向三：报告生成前移，而不是只做报告事后质控
- 从商业价值看，事后找错的价值通常低于事中辅助。
- 如果 V3 的统一语义层做出来，最值得延展的不是“查错更多”，而是：
  - 报告书写过程中的实时提醒
  - 部位/结论一致性即时提示
  - 申请单与报告不符的即时提示
  - 模板推荐和表达归一化建议
- 这会把系统从“质控工具”变成“医生工作流助手”，使用频率和用户感知价值都会更高。

### 方向四：多院区数据治理与科研数据资产平台
- 放射报告一旦实现高质量结构化，就可以支撑：
  - 多院区科研队列构建
  - 病种检索和病例回顾
  - 质控运营看板
  - 疾病谱和检查质量分析
  - 教学与住培反馈
- 这条路径的价值不在前端“质控结果”，而在后端“可计算数据资产”。
- 对大型医院集团、科研型医院和区域平台，这条路线的商业想象空间通常大于单点 SaaS 质控。

## 最值得优先验证的商业切口

### 第一优先：多院区接入成本下降
- 这是最容易证明、最容易签单、也最符合当前技术路线的价值点。
- 验证指标不应只看模型精度，还要看：
  - 新院区从接入到可用所需时间
  - 人工规则适配工作量
  - 新增术语维护成本
  - 质控覆盖任务数

### 第二优先：结构化标准化能力是否可复用
- 需要验证 V3 是否真的能把不同院区的异构表达映射到统一结构语义层。
- 如果这一点成立，后续质控、科研、随访、教学、运营都能复用这层能力。
- 这是整个系统最强的长期价值杠杆。

### 第三优先：前移到报告书写工作流的能力
- 如果只做事后审核，使用频率和价值感知都有限。
- 如果能前移到书写环节，成为实时辅助系统，其商业价值和产品粘性会明显提升。

## 商业化定位建议

### 不建议的定位
- 不建议把 V3 定位为“一个更复杂的放射质控模型”。
- 这种定位很容易把系统价值压缩成低频错误检测，难以支撑复杂系统投入。

### 更合理的定位
- 建议把 V3 定位为“面向多院区泛化的放射语义标准化与质控中台”。
- 其中包含三层价值：
  1. 底层：解剖实体与医学语义标准化
  2. 中层：多院区泛化的规则/模型协同推理能力
  3. 上层：质控、书写辅助、数据治理、科研和运营分析

### 商业化顺序建议
1. 先以“多院区质控快速接入”切入，证明泛化价值。
2. 再扩展到“结构化标准化引擎”，输出中台能力。
3. 最后前移到“书写辅助”和后扩到“科研/治理平台”。

## 对 V3 研发目标的反向修正
- V3 的核心研发目标不应只是“把更多任务模型化”。
- 更合理的目标应是：
  1. 建立多院区通用的结构化语义层
  2. 降低新院区接入的人力适配成本
  3. 让同一套能力可以同时服务质控、辅助书写和数据治理
- 如果 V3 只能提升一点质控精度，却不能沉淀成跨院区可复用能力，那么商业价值会明显低于其研发复杂度。

## 产品策略版

### 一句话定位
- V3 不是“更复杂的放射质控模型”，而是“面向多院区泛化的放射语义标准化与质控中台”。

### 目标客户

#### 第一目标客户
- 医联体和医院集团
- 多院区统一管理的三甲医院
- 区域影像中心
- 第三方影像平台或影像云平台

#### 第二目标客户
- 专科医院集团
- 科研型医院
- 有统一数据治理需求的放射科中心

#### 暂不优先的客户
- 单院区、单科室、报告风格长期稳定且已有成熟规则体系的医院
- 对这类客户，V1/V2 已经足够，V3 并不是最高优先级产品

### 客户核心痛点

#### 显性痛点
- 新院区接入慢，每接一个院区都要重新补规则、补字典、调阈值
- 多院区质控口径不一致，难以统一管理
- 规则维护高度依赖少数熟悉业务和配置的人
- 新术语、新模板、新缩写出现后，系统响应慢

#### 隐性痛点
- 放射报告无法形成稳定、统一、可复用的结构化数据资产
- 各院区报告风格差异导致科研、检索、随访和运营统计口径混乱
- 现有质控系统更多是“工具”，而不是“平台能力”

### 核心价值主张

#### 对管理者的价值
- 缩短新院区上线周期
- 降低规则维护和适配成本
- 提升多院区质控标准一致性
- 形成可运营、可分析、可复用的放射数据资产

#### 对放射科的价值
- 提供跨院区风格下更稳定的质控结果
- 降低模板差异带来的误报和漏报
- 为后续书写辅助、模板优化和教学反馈提供基础能力

#### 对平台方的价值
- 获得可复制的多院区接入能力，而不是项目制逐院定制
- 形成中台化能力，可向质控、结构化、科研、治理等多个模块复用

### 产品切入点

#### 第一切入点：多院区质控快速接入
- 这是最容易商业化验证的入口。
- 客户最容易感知的价值不是“模型多先进”，而是：
  - 新院区接入更快
  - 人工适配更少
  - 质控结果更统一
  - 规则维护压力更低

#### 第二切入点：放射报告标准化引擎
- 当系统具备稳定的跨院区语义统一能力后，可把能力包装成独立中台服务：
  - 解剖部位标准化
  - 描述语义归一化
  - 结论结构化
  - 术语同义映射

#### 第三切入点：书写过程实时辅助
- 比起事后找错，事中辅助的使用频率更高、用户感知价值更强。
- 在此基础上可以延展：
  - 实时一致性提醒
  - 申请单对照提醒
  - 术语归一化建议
  - 模板推荐和书写提示

### MVP 定义

#### MVP 目标
- 不追求一次性覆盖所有质控任务。
- 只验证一个关键命题：V3 是否能显著降低多院区接入成本，并在跨院区场景下优于直接复制 V2 规则。

#### MVP 建议范围
- 覆盖 3 个以上风格差异明显的院区
- 聚焦 4 类最能体现泛化能力的任务：
  - conclusion_missing
  - orient_error
  - contradiction
  - apply_orient
- 实体层同步验证“规则主导 + 模型补充”的标准化能力

#### MVP 不做的事
- 不要求替换所有 V2 规则项
- 不要求一开始就做完整科研平台
- 不要求一开始就做全量书写辅助

### MVP 成功标准
- 新院区接入周期明显下降
- 新增术语/表达的人工维护量明显下降
- 跨院区验证集上的核心任务表现明显优于 V2 默认复制方案
- 至少形成一套可复用的结构化语义中间层输出

### ROI 验证方式

#### 不建议只看模型指标
- 产品价值不能只用 precision、recall、F1 来证明。
- 对客户更重要的是部署效率和维护成本。

#### 建议重点跟踪的 ROI 指标
1. 新院区从接入到可用的时间
2. 单院区新增规则/词典维护工时
3. 多院区统一部署后的人工运营成本
4. 关键任务的跨院区效果稳定性
5. 是否形成可复用的结构化数据资产输出

### 商业化模式建议

#### 模式一：多院区质控中台项目制 + 年服务费
- 适合医院集团、区域平台、影像中心
- 第一阶段最现实

#### 模式二：标准化语义引擎授权
- 将实体标准化、语义归一化、结构化输出能力作为底层引擎输出
- 适合平台厂商和生态合作伙伴

#### 模式三：工作流辅助增值模块
- 在质控之外，延展到实时辅助书写、模板优化、运营分析等模块
- 提升单客户价值和产品粘性

### 产品演进路径

#### 阶段一：卖“多院区接入效率”
- 核心卖点是降本增效
- 客户愿意付费的理由是接入更快、维护更轻

#### 阶段二：卖“统一语义中台能力”
- 核心卖点是标准化和可复用
- 产品从工具升级为中台

#### 阶段三：卖“工作流与数据资产价值”
- 核心卖点是高频使用和平台复用
- 包括书写辅助、科研、治理、运营分析

### 竞争壁垒
- 不是单个模型参数规模
- 不是单次测试分数
- 真正的壁垒来自三件事：
  1. 多院区真实放射报告数据和持续数据生产能力
  2. 实体标准化与知识图谱锚定能力
  3. 规则、领域模型、强 LLM 复核三者协同形成的闭环

### 最终产品判断
- 如果 V3 只是一个“更准一点的质控工具”，商业天花板有限。
- 如果 V3 能成为“多院区放射语义标准化与质控中台”，则它的价值将不止于质控，而会延展到数据治理、书写辅助、科研和平台化能力。
- 因此，产品策略上应坚持一条主线：
  先用多院区质控切入，验证接入效率与泛化能力；
  再把底层结构化语义能力沉淀成可复用中台；
  最终扩展到更高频、更高价值的工作流和数据资产场景。
